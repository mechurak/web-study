# 00. Overview — 웹 서비스 아키텍처 큰 그림

> **이 챕터를 읽고 나면 답할 수 있어야 하는 질문**
> - 브라우저에 URL을 입력하면 응답이 오기까지 무슨 일이 일어나는가?
> - "프론트엔드 / 백엔드 / DB"의 경계는 어디이고, 요즘 스택에서는 왜 흐려지는가?
> - 이 레포의 스택(Next.js + Vercel + Supabase + NestJS)은 각각 어느 레이어를 담당하는가?

---

## 1. 요청의 일생 (Life of a Request)

`https://myapp.com/memos`를 브라우저에 입력했을 때 일어나는 일:

```
브라우저
  │ ① DNS 조회: myapp.com → IP 주소 (실제로는 CDN의 엣지 서버 IP)
  │ ② TCP 연결 + TLS 핸드셰이크 (https의 자물쇠)
  │ ③ HTTP 요청: GET /memos
  ▼
CDN / Edge (전 세계에 흩어진 서버, 유저와 가장 가까운 곳이 응답)
  │ ④ 캐시에 있는가?
  │    ├─ 있음(정적 파일, SSG 페이지) → 즉시 응답. 오리진까지 안 감 ★가장 빠른 경로
  │    └─ 없음 → 오리진 서버로 전달
  ▼
오리진 서버 (Vercel이라면 serverless function)
  │ ⑤ 서버 코드 실행: 세션 확인, 데이터 필요하면 ↓
  ▼
DB (Supabase Postgres)
  │ ⑥ SQL 실행, 결과 반환
  ▼
오리진 서버
  │ ⑦ 데이터를 HTML(SSR) 또는 JSON(API)으로 만들어 응답
  ▼
브라우저
    ⑧ HTML 파싱 → CSS/JS/이미지 추가 요청 (①~④ 반복) → 렌더링
    ⑨ JS 로드 후 hydration → 이후 클릭/입력은 API 호출(JSON)로 처리
```

### 여기서 나오는 핵심 구분 3가지

| 구분 | 무엇을 받나 | 누가 만드나 | 예 |
|---|---|---|---|
| **정적 파일 서빙** | 미리 만들어진 HTML/JS/CSS | 빌드 타임에 생성, CDN이 서빙 | 블로그 글, 랜딩 페이지 |
| **서버 렌더링(SSR)** | 요청 순간에 만든 HTML | 오리진 서버가 요청마다 생성 | 로그인한 유저의 대시보드 |
| **API 호출** | JSON 데이터 | 서버가 생성, 브라우저 JS가 화면에 반영 | 무한스크롤, 폼 제출 |

한 페이지 안에서도 세 가지가 섞인다. 첫 화면은 SSR로 받고, 이후 상호작용은 API 호출로 처리하는 식.
**"이 요청은 셋 중 무엇이고, 어디까지 갔다 오는가?"** 를 항상 그릴 수 있으면 성능 문제(왜 느리지?)와 아키텍처 결정(어디서 렌더링하지?)의 대부분이 풀린다.

## 2. 레이어별 역할과 이 레포의 스택

전통적인 3계층 구분과 우리 스택의 대응:

| 레이어 | 전통적 역할 | 이 레포에서 | 배포되는 곳 |
|---|---|---|---|
| **Frontend** | 화면, 사용자 상호작용 | **Next.js** (+ Tailwind, shadcn/ui) | **Vercel** |
| **Backend** | 비즈니스 로직, 인증/인가, 외부 연동 | **NestJS** | Railway/Fly.io 등 (Vercel 아님!) |
| **DB** | 데이터 저장, 무결성 | **Supabase** (Postgres) | Supabase 클라우드 |

### 그런데 경계가 흐리다 — 이게 현대 스택의 특징

교과서적 3계층이 그대로 안 지켜지는 지점들:

- **Next.js는 프론트만이 아니다.** 서버 컴포넌트·Server Action·Route Handler로 서버 코드를 실행한다. 간단한 서비스는 Next.js만으로 백엔드까지 끝난다.
- **Supabase는 DB만이 아니다.** 인증(회원가입/로그인), 파일 스토리지, 실시간 구독까지 제공한다. 원래 백엔드가 하던 일의 절반을 가져갔다. 심지어 브라우저가 DB에 **직접** 쿼리한다(RLS가 지켜줌, 02-supabase 03장).
- **그래서 NestJS는 "필요해지면" 등장한다.** 복잡한 비즈니스 로직, 트랜잭션, 배치 작업, 외부 시스템 연동 — Next.js와 Supabase의 틈이 벌어질 때 그 사이를 메운다.

이 레포의 학습 순서(01→02→03)가 바로 이 순서다: **Next.js와 Supabase로 어디까지 되는지 먼저 체감하고, 부족한 지점에서 NestJS를 도입한다.** "백엔드는 원래 있어야 하니까"가 아니라 "이래서 필요하구나"를 겪는 게 목표.

## 3. 왜 이 스택인가 — 대안 비교

### Next.js vs 대안들

| | Next.js | SPA (Vite + React) | Remix / 기타 |
|---|---|---|---|
| 첫 화면 | 서버에서 HTML 완성 → 빠름 | 빈 HTML + JS 로드 후 렌더 → 느림 | 서버 렌더 (Next와 유사) |
| SEO | 좋음 (크롤러가 완성된 HTML을 봄) | 나쁨 (추가 조치 필요) | 좋음 |
| 백엔드 겸용 | Route Handler, Server Action | 없음 (별도 API 서버 필수) | 있음 |
| 러닝커브/생태계 | 무겁지만 생태계 최대 | 단순함 | 좋은 설계지만 생태계 작음 |

- **SPA가 여전히 맞는 경우**: 로그인 뒤에서만 쓰는 내부 도구, 관리자 페이지. SEO와 첫 로딩이 안 중요하면 SPA가 더 단순하다.
- **Next.js를 고르는 이유**: 공개 서비스(SEO 필요) + 한 프레임워크로 프론트/백 겸용 + Vercel과의 통합 + 채용시장/자료의 압도적 규모.

### Supabase vs 대안들

| | Supabase | Firebase | 직접 Postgres 운영 |
|---|---|---|---|
| DB 종류 | **Postgres (SQL, 관계형)** | Firestore (NoSQL) | Postgres |
| 쿼리 능력 | JOIN, 트랜잭션, SQL 전부 | 제한적 (JOIN 없음) | 전부 |
| 부가 기능 | 인증/스토리지/실시간 내장 | 인증/스토리지/실시간 내장 | 전부 직접 구축 |
| 락인 | 낮음 (표준 Postgres, 오픈소스 — 언제든 이사 가능) | 높음 (독점 API) | 없음 |
| 운영 부담 | 없음 (백업/패치/스케일링 해줌) | 없음 | 전부 내 몫 |

- **Supabase를 고르는 이유**: "그냥 Postgres"라서. SQL을 배우면 그대로 자산이 되고, 서비스가 커져서 이사 가도 DB는 표준이다. 여기에 인증·스토리지가 공짜로 딸려 와서 초기 개발 속도는 Firebase급.
- **직접 운영이 맞는 경우**: 규제로 데이터 위치를 통제해야 하거나, 규모가 커져 관리형 비용이 역전될 때. 학습·초기 서비스에선 시기상조.

### NestJS vs 대안들

| | NestJS | Express | "Next.js API로 다 하기" |
|---|---|---|---|
| 구조 | 모듈/DI/레이어 **강제** | 자유 (= 무질서 가능) | 파일 라우팅뿐 |
| 규모 확장 | 팀·코드 커져도 일관성 유지 | 팀의 규율에 의존 | 로직 커지면 한계 |
| 내장 기능 | 검증, 인증 가드, 테스트, 문서화 | 전부 라이브러리 조합 | 최소한 |
| 실행 모델 | 상시 서버 (웹소켓, 크론, 큐 가능) | 상시 서버 | serverless (제한적) |

- **NestJS를 고르는 이유**: 구조를 강제해서 "어디에 뭘 놓을지" 고민이 사라지고, 팀이 커져도 코드가 비슷하게 생겼다. Spring 경험자에게 익숙한 구조라 백엔드 설계 관례(레이어드 아키텍처, DI)를 배우는 교재로도 좋다.
- **Express가 맞는 경우**: 작은 마이크로서비스 하나, 프로토타입. 강제된 구조가 오히려 짐일 때.

### Vercel vs 대안들

| | Vercel | AWS 직접 구성 | VPS 한 대 |
|---|---|---|---|
| 배포 | git push = 배포 끝 | CDN/서버/CI 직접 조립 | 직접 스크립트 |
| 프리뷰 배포 | PR마다 자동 | 직접 구축 | 없음 |
| Next.js 최적화 | 제작사라 완벽 (ISR, 엣지) | 직접 재현해야 함 | 제한적 |
| 비용 | 무료 시작, 커지면 비쌈 | 초반 복잡, 규모에선 저렴 | 저렴하지만 손이 감 |
| 제약 | serverless 제약 (실행시간 제한, 상시 프로세스 불가) | 없음 | 없음 |

- **Vercel을 고르는 이유**: 배포·CDN·CI·프리뷰가 전부 자동. 인프라 공부 없이 "서비스 만들기"에 집중할 수 있다. 단, **상시 서버가 필요한 NestJS는 못 올린다** — 그래서 04-deployment에서 Nest 호스팅을 따로 다룬다.

## 4. 목표 아키텍처

capstone에서 실제로 구현할 최종 형태:

```
                        ┌─────────────────────────────┐
                        │          브라우저             │
                        └──────┬──────────────────────┘
                               │ HTTPS
                        ┌──────▼──────────────────────┐
                        │   Vercel (CDN + Edge)        │
                        │   Next.js                    │
                        │   - 정적/SSG 페이지 → CDN     │
                        │   - SSR/Server Action → 함수 │
                        └──┬────────────────────┬─────┘
              단순 CRUD,    │                    │  복잡한 로직,
              인증, 실시간   │                    │  트랜잭션, 배치
                        ┌──▼──────────┐   ┌─────▼─────────────┐
                        │  Supabase    │   │  NestJS API 서버   │
                        │  - Auth(JWT) │◄──┤  (Railway 등)      │
                        │  - RLS가 방어 │   │  - JWT 검증(Guard) │
                        │  - Storage   │   │  - 인가는 코드가 방어│
                        │  - Realtime  │   └─────┬─────────────┘
                        │      │       │         │ Prisma (pooler)
                        │  Postgres ◄──┼─────────┘
                        └──────────────┘
```

**데이터 경로가 2개인 것이 이 아키텍처의 핵심 결정이다:**

| 경로 | 언제 쓰나 | 누가 보안을 지키나 |
|---|---|---|
| Next → Supabase 직접 | 단순 CRUD, 인증, 파일, 실시간 | **RLS** (DB의 행 단위 정책) |
| Next → NestJS → Postgres | 트랜잭션, 복잡한 검증/계산, 외부 연동 | **NestJS의 인가 로직** (Guard + 소유권 검사) |

인증은 한 곳(Supabase Auth)에서만 한다. 브라우저는 Supabase에서 JWT를 받고, NestJS는 그 JWT의 서명만 검증해서 유저를 신뢰한다(03-nestjs 04장). 비밀번호와 세션을 NestJS가 다시 관리할 필요가 없다.

## 직접 해보기

- [ ] **아무 사이트나 열고 DevTools Network 탭 관찰하기** — 첫 HTML 응답을 클릭해 내용 확인(완성된 HTML이면 SSR/정적, 거의 빈 HTML이면 SPA), 이후 이어지는 fetch/XHR 요청들(API 호출)을 구분해보기.
  → **의미**: 1장의 "정적 서빙 / SSR / API 호출" 구분은 표로 외우는 게 아니라 Network 탭에서 눈으로 식별할 수 있어야 한다. 이게 되면 아무 사이트나 열어도 "이 서비스는 어떤 렌더링 전략을 쓰는구나"를 읽어낼 수 있고, 내 서비스가 느릴 때 어느 구간(HTML? API? 리소스?)이 문제인지 좁힐 수 있다.

- [ ] **응답 헤더에서 `cache-control`, `x-vercel-cache`(Vercel 사이트라면) 찾아보기** — 값이 `HIT`이면 CDN 캐시에서, `MISS`면 오리진까지 갔다 온 것.
  → **의미**: "CDN 캐시에 있으면 오리진까지 안 간다"(요청의 일생 ④)가 실제로 헤더에 기록된다는 것을 확인하는 실습. 같은 페이지를 새로고침하며 MISS → HIT으로 바뀌는 걸 보면 캐싱이 추상 개념이 아니라 관찰 가능한 동작임을 체감한다. 04-deployment에서 Vercel 캐시를 다룰 때 이 헤더를 계속 보게 된다.

- [ ] **`dig 도메인` 또는 https://dnschecker.org 로 DNS 조회 결과 확인하기** — 유명 서비스의 도메인을 조회하면 최종적으로 CDN 업체(Cloudflare, Vercel 등)의 IP가 나오는 것 확인.
  → **의미**: "요청은 서비스의 서버가 아니라 CDN 엣지에 먼저 도착한다"(요청의 일생 ①)를 확인하는 실습. 나중에 04-deployment에서 커스텀 도메인을 연결할 때 하는 작업이 정확히 이것(내 도메인이 Vercel을 가리키게 DNS 레코드 설정)이라서, 미리 해보면 그 챕터가 쉬워진다.

- [ ] **위 목표 아키텍처를 안 보고 다시 그려보기** — 두 데이터 경로와 각각의 보안 담당(RLS vs NestJS 인가 로직)까지 포함해서.
  → **의미**: 이 그림이 앞으로 모든 챕터의 지도다. 각 챕터는 이 그림의 한 조각을 만드는 과정이고(01=Vercel 상자, 02=Supabase 상자, 03=NestJS 상자, 04=상자 사이의 배포), 특히 "두 경로 중 어느 쪽을 쓸까"는 capstone 설계에서 실제로 반복하게 될 판단이다. 안 보고 그릴 수 있다 = 큰 그림이 머리에 있다는 검증.
